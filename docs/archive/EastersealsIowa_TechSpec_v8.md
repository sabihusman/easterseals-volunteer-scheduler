## 1. Executive Summary

A web and mobile-friendly volunteer scheduling platform for Easterseals Iowa. Three distinct user roles (Volunteer, Coordinator, Admin) collaborate around a normalized PostgreSQL data model hosted on Supabase. The frontend is a Vite + React + TypeScript SPA deployed on Vercel. All business logic lives in the database layer as SQL triggers, RLS policies, and SECURITY DEFINER functions, with a thin React UI on top.

### Core capabilities
- **Volunteer booking** with waitlists, 2-hour accept windows, and slot selection
- **Department-scoped coordinator management** with fine-grained RLS
- **Automated shift reminders** (pg_cron) — 24h, 2h before, and unactioned follow-ups
- **Admin shift lifecycle** — cancel, hard-delete, action-off, reconcile
- **Check-in window** (30 minutes before start) with early-time credit
- **Self-service shift confirmation** with hours + star rating
- **Retroactive hour correction** by coordinators / admins
- **Messaging** with per-user local delete + admin messaging block
- **Reports + analytics** for coordinators and admins
- **Volunteer invites** (token-based)
- **Document compliance** (BG checks, expiry tracking)
- **MFA TOTP enrollment** with admin reset
- **Calendar export** (ICS + Google Calendar links)

## 3. Authentication & Authorization

### 3.1 Auth flow
1. **Email/password** via Supabase GoTrue with Turnstile captcha
2. **Username sign-in** — `get_email_by_username(p_username)` RPC resolves username to email, then normal password flow; generic "Invalid credentials" error message to avoid username enumeration
3. **Google OAuth** via Supabase redirect flow
4. **Forgot password** with Resend email delivery + captcha
5. **MFA TOTP** optional per-user enrollment, factor ID stored with profile
6. **Session refresh** via Supabase client automatic refresh
7. **Sign out** clears `localStorage["sb-*"]` keys

### 3.2 Role resolution
`AuthContext` (`src/contexts/AuthContext.tsx`):
- Listens to `supabase.auth.onAuthStateChange`
- Defers profile fetch via `setTimeout(..., 0)` to avoid the GoTrue auth-lock deadlock
- Exposes `{ user, profile, role, loading, signOut, refreshProfile }`

`ProtectedRoute` (`src/App.tsx`):
- `if (loading) return <Loading>`
- `if (!user) return <Navigate to="/auth">`
- `if (requiredRole && !role) return <Loading>` — wait for profile
- `if (requiredRole && !requiredRole.includes(role))` — redirect to appropriate home
- Else render children wrapped in `AppLayout`

### 3.3 RLS policy philosophy
- **Every table has RLS enabled**
- **Permissive policies** are OR'd (PostgreSQL semantics) — any match grants access
- **Helper functions** for common checks:
  - `is_admin()` — SECURITY DEFINER, reads profiles directly
  - `is_coordinator_or_admin()` — SECURITY DEFINER
  - `has_active_booking_on(shift_id)` — SECURITY DEFINER (avoids policy recursion)
- **Critical rule**: any policy that references another table whose own policies reference back MUST use a SECURITY DEFINER helper to break the cycle. The session discovered this the hard way — see §10 Incident Log.

## 4. Business Logic (DB-Resident)

### 4.1 Trigger chain on `shift_bookings` INSERT
The order matters. Triggers fire alphabetically within the BEFORE/AFTER groups.

```
BEFORE INSERT:
  1. trg_enforce_dept_restriction
       └─ Reject if volunteer is restricted from the shift's department
  2. trg_enforce_shift_not_ended_insert
       └─ Reject if shift_end_at() <= now()
  3. trg_prevent_overlapping_bookings
       └─ Reject if volunteer already confirmed on a time-overlapping shift
  4. trg_validate_booking_slots
       └─ WHEN (NEW.booking_status = 'confirmed')
       └─ Lock shifts row, count confirmed, demote to waitlisted if full
  5. trg_volunteer_only_booking
       └─ Reject coordinators/admins booking as volunteers

AFTER INSERT:
  6. trg_sync_slots
       └─ SECURITY DEFINER — increments shifts.booked_slots
       └─ Also flips shifts.status to 'full' via update_shift_status
```

### 4.2 Trigger chain on `shift_bookings` UPDATE OF booking_status
```
BEFORE UPDATE:
  1. trg_enforce_shift_not_ended_update
       └─ Exempt for confirmed→confirmed (e.g. action-off updates
          other columns without changing booking_status)
  2. trg_prevent_overlapping_bookings (same as INSERT)
  3. trg_validate_booking_slots_update
       └─ WHEN (OLD DISTINCT FROM 'confirmed' AND NEW = 'confirmed')
       └─ Re-check capacity on waitlisted→confirmed and cancelled→confirmed

AFTER UPDATE OF booking_status:
  4. trg_sync_slots
       └─ Handles all transitions: confirmed↔cancelled, waitlisted↔confirmed,
          cancelled→confirmed (re-activation), confirmed→waitlisted (demotion)
  5. trg_waitlist_promote
       └─ WHEN (OLD='confirmed' AND NEW='cancelled')
       └─ Calls promote_next_waitlist(shift_id)

AFTER UPDATE OF confirmation_status:
  6. trg_recalc_consistency / trg_recalculate_consistency
       └─ Recomputes profiles.consistency_score + extended_booking flag
  7. trg_recalculate_points_update
       └─ Recomputes profiles.volunteer_points
```

### 4.3 Trigger chain on `shift_bookings` DELETE
```
AFTER DELETE:
  1. trg_recalculate_consistency_delete
  2. trg_recalculate_points_delete
  3. trg_waitlist_promote_delete
       └─ WHEN (OLD.booking_status = 'confirmed')
       └─ Promote next waitlisted for the same shift
  4. trg_cleanup_notifications_on_booking_delete
       └─ Removes orphan notifications that referenced this booking
```

### 4.4 Slot sub-bookings (`shift_booking_slots`)
```
AFTER INSERT OR DELETE:
  1. trg_sync_slot_count → sync_slot_booked_count()
       └─ SECURITY DEFINER — clamps booked_slots to [0, total_slots]
```

## 8. Key User Flows

### 8.1 Volunteer books a shift
1. User on `/shifts`; query filters to `shift_date >= today`, `status IN ('open','full')`, within booking window (14 days standard, 21 days extended), not in restricted departments, not hidden by BG check
2. Client-side: filters out shifts whose `end_time` has passed, shifts already in user's bookingIds set
3. User clicks **Book Shift** → `SlotSelectionDialog` opens keyed by shift id
4. Dialog fetches `shift_time_slots` for the shift
5. User selects one or more slots (row-level click toggles selection)
6. Dialog does a fresh COUNT to verify capacity; if full, native `window.confirm` prompts to join waitlist
7. If existing cancelled booking: UPDATE it back to `confirmed`/`waitlisted` (re-activation path)
8. Else: INSERT new `shift_bookings` row
9. Trigger chain (§4.1) fires: dept check, end check, overlap check, capacity check → demote to waitlist if full, then AFTER trigger increments counter
10. Dialog reads back `booking_status` from DB to determine actual stored value (in case of demotion)
11. Insert `shift_booking_slots` rows for selected slots; sync trigger increments slot counters
12. Fire-and-forget booking confirmation email via `sendEmail`
13. `onBooked()` → parent `fetchData()` refreshes list and carousel

### 8.2 Waitlist promotion
1. Volunteer A cancels confirmed booking (UPDATE `booking_status='cancelled'`)
2. `trg_sync_slots` decrements counter
3. `trg_waitlist_promote` fires → `promote_next_waitlist(shift_id)`
4. Function verifies shift is not cancelled and not ended
5. Locks the oldest waitlisted row with `FOR UPDATE SKIP LOCKED`
6. Sets `waitlist_offer_expires_at = now() + 2 hours`
7. INSERTs `notifications` row with `type='waitlist_offer'` and `data.booking_id`
8. `notify_email_on_notification` trigger fires → POST to `notification-webhook` edge function
9. Edge function reads profile's notification prefs, routes to `send-email` / `send-sms` as appropriate
10. Volunteer B sees amber "Waitlist spot opened" card on dashboard with Accept / Decline buttons and live countdown
11. B clicks **Accept Shift** → `waitlist_accept(booking_id)` RPC
12. RPC re-validates: not expired, shift not cancelled, shift not ended
13. UPDATE `booking_status='confirmed'`, clears `waitlist_offer_expires_at`
14. `trg_validate_booking_slots_update` re-checks capacity (defense in depth)
15. `trg_sync_slots` increments counter; `trg_recalculate_*` updates stats
16. Insert "Shift confirmed" notification

### 8.3 Check-in window
- Volunteer dashboard computes `shiftStartMs` in local time
- Check In button rendered only when `now >= start - 30min && now <= end`
- `handleCheckIn` re-validates and rejects with a toast if outside window
- Early check-in credit: `ShiftConfirmation.maxHours` adds up to 30 min of pre-start time if `checked_in_at < start_time`

### 8.4 Shift confirmation (after shift ends)
1. Cron `unactioned-shift-volunteer-reminder` inserts up to 2 reminder notifications
2. Volunteer clicks email CTA → `/my-shifts/confirm/:bookingId`
3. Form: "Did you attend?" → Yes/No
4. Yes path: hours input (capped at shift duration + early check-in credit) + 1-5 star rating + optional text feedback
5. No path: marked as no-show, notifies coordinator (type `booking_changed`)
6. Upsert into `volunteer_shift_reports`
7. `trg_sync_volunteer_hours` triggers `resolve_hours_discrepancy` to merge volunteer vs coordinator hours if both reported
8. `trg_recalculate_points` and `trg_recalculate_consistency` update profile stats

### 8.5 Admin action-off
1. Admin on `/admin/unactioned-shifts` sees oldest-first list via `get_unactioned_shifts()`
2. Clicks **Action Off (Confirm)** → confirm dialog
3. `admin_action_off_shift(booking_id)` RPC:
   - Verifies auth
   - Computes duration from shift start/end (+30m grace for early check-in)
   - Upserts `volunteer_shift_reports` with `self_confirm_status='attended'`
   - UPDATEs `shift_bookings` to `confirmation_status='confirmed'`, `final_hours=duration`, `hours_source='admin_action_off'`
   - Inserts "Shift marked complete" notification
4. Triggers recalc points + consistency
5. Shift disappears from Unactioned Shifts list

### 8.6 Retroactive hour correction
1. Coordinator on `/coordinator` sees confirmed bookings
2. Clicks pencil icon next to hours badge → Edit Hours dialog
3. `admin_update_shift_hours(booking_id, hours)` RPC:
   - Auth check
   - Clamps at max `duration + 30m`
   - UPDATE booking: `final_hours`, `coordinator_reported_hours`, `hours_source='coordinator'`
   - Recomputes `profiles.total_hours` directly
   - Calls `recalculate_points()`

## 9. Feature-level details

### 9.1 Messaging with local delete
- `conversations` + `conversation_participants (cleared_at)` + `messages`
- "Delete conversation" sets `cleared_at = now()` on the user's participant row only — other participants unaffected
- `ConversationList` and `ConversationThread` filter messages by `created_at > cleared_at`
- `get_unread_conversation_count()` uses `GREATEST(last_read_at, cleared_at)` as the cutoff
- `useUnreadCount` hook subscribes to realtime INSERTs on `messages` for live badge updates

### 9.2 Admin messaging block
- `profiles.messaging_blocked` boolean
- RLS policies on `messages` INSERT and `conversations` INSERT include `NOT EXISTS (... messaging_blocked=true)`
- Blocked users can still READ (so they see replies from admins)
- Client-side reinforcement: banner on `/messages`, disabled send button, disabled compose buttons

### 9.3 Coordinator department scoping
- Coordinators can only CRUD shifts in their assigned departments (via `department_coordinators`)
- `ManageShifts` fetches only assigned departments for the department dropdown
- RLS policy `shifts: coord/admin insert` and `shifts: coord/admin update` enforce dept membership at DB level
- Client-side `handleSave` pre-validates selection against allowed list
- `CoordinatorDashboard` dropdown includes "All My Departments" for coordinators with 2+ assignments

### 9.4 Shift report privacy
- Individual ratings NEVER exposed to coordinators/admins
- `get_shift_rating_aggregates()` and `get_department_report()` enforce `HAVING COUNT >= 2` so small samples can't be reverse-engineered
- `volunteer_shift_reports_safe` view excludes `star_rating` and `shift_feedback` columns for coordinator hours reconciliation queries

### 9.5 Consistency score
- Computed from rolling last 5 `confirmation_status IN ('confirmed','no_show')` bookings for shifts where `shift_date <= today`
- `consistency_score = round(attended / total * 100, 2)`
- `extended_booking = (score >= 90 AND total >= 5)`
- Updated by `trg_recalculate_consistency` on every confirmation_status change

### 9.6 Points
- `shift_pts` = SUM(final_hours) * 10 on confirmed completed shifts
- `rating_pts` = COUNT(5-star ratings) * 5
- `milestone_pts` = FLOOR(total_hours / 10) * 25
- Total = shift + rating + milestone

### 9.7 Username sign-in
- `profiles.username` is citext UNIQUE (case-insensitive)
- `get_email_by_username(p_username)` RPC is public-exec; returns email or NULL
- `Auth.tsx` login handler: if identifier doesn't contain `@`, call RPC to resolve to email, then normal password flow
- Generic "Invalid credentials" error message on lookup failure to avoid username enumeration

### 9.8 Calendar export
- `calendar-utils.ts` `generateICS(shift)` produces RFC 5545 compliant output with UID, DTSTAMP, TZID (America/Chicago), and RFC-escaped special chars
- `googleCalendarUrl(shift)` builds a `https://calendar.google.com/calendar/render?action=TEMPLATE&...` URL
- Dashboard and shift detail views expose both as buttons
- Personal feed URL via `calendar-feed` edge function (auth via token)

## 10. Incident Log — v7 → v8 (this QA session)

Nine critical bugs found and fixed during extensive live testing. Full details in session commits.

### P0 bugs (total outages, all FIXED)

1. **Infinite RLS recursion** (`1ca12ea`) — `shifts: read booked` policy created a cycle through `shift_bookings` → `profiles`. Every authenticated profile fetch returned HTTP 500. App was stuck on "Loading…" for everyone. **Fix**: SECURITY DEFINER helper `has_active_booking_on()`.

2. **`notify_email_on_notification` trigger called nonexistent function** (`249ee79`) — trigger was calling `extensions.http_post()` instead of `net.http_post()`. Every notification INSERT from a user with `notif_email=true` raised `42883` and rolled back the parent transaction. Silently broke: admin_action_off_shift, admin_delete, waitlist_accept, waitlist_decline, promote_next_waitlist, admin hard-delete, self no-show, and probably more. **Fix**: corrected signature + wrapped in EXCEPTION block so webhook failures can never break the parent transaction.

### P1 bugs (high severity, all FIXED)

3. **`validate_booking_slot_count` double-incrementing** (`93b4b41`) — BEFORE INSERT trigger did `UPDATE shifts SET booked_slots = actual_booked + 1` but AFTER trigger `sync_booked_slots` also incremented. Every booking counted twice. **Fix**: removed redundant UPDATE.

4. **`sync_booked_slots` missed `cancelled → confirmed` branch** (`93b4b41`) — re-booking after cancel silently left counter below ground truth. **Fix**: widened branch condition.

5. **`promote_next_waitlist` ignored shift validity** (`93b4b41`) — admin cancel-shift triggered promotion into the just-cancelled shift. **Fix**: verify `status <> 'cancelled'` and `shift_end_at() > now()`.

6. **`waitlist_accept` didn't re-validate shift** (`93b4b41`) — could accept after shift cancelled or ended. **Fix**: added pre-checks.

7. **`useUnreadCount` N+1 query flood** (`ed3148a`) — one HEAD request per conversation, on every mount + every realtime event. Caused pooler 503s that cascaded through Promise.all blocks. **Fix**: new `get_unread_conversation_count()` RPC.

8. **HEAD count on profiles returned 503** (`ed3148a`) — Admin Dashboard stats used `count=exact, head=true` which triggers full RLS scan. **Fix**: light `select("id")` and client-side length.

9. **Date display off-by-one** (`ed3148a`) — `new Date("2026-04-08")` parses as UTC midnight, displays as previous day west of UTC. Affected every shift date display across 14 call sites. **Fix**: `parseShiftDate()` helper that anchors to local midnight.

### P1 bug discovered AND root-caused THIS session

10. **`sync_booked_slots` silently blocked by RLS** (`5f5c76b`) — THE ACTUAL root cause of the counter-drift symptoms that led me to chase several dead ends. The trigger ran `UPDATE public.shifts SET booked_slots = ...` in the caller's role context. `shifts` has an UPDATE RLS policy `shifts: coord/admin update` which only permits coord/admin. When a **volunteer** booked a shift, the trigger's UPDATE was silently rejected (zero rows affected, no error raised), leaving `shifts.booked_slots` at its old value. Live-verified via direct REST INSERT as an authenticated volunteer: 201 Created but counter stayed at 0. Direct SQL INSERT (superuser, bypasses RLS) incremented correctly. **Fix**: marked `sync_booked_slots` and `sync_slot_booked_count` SECURITY DEFINER. Same issue affected `shift_time_slots` which has no UPDATE policy at all.

### P2 bugs fixed this session
11. **Slot checkbox double-toggle** (`b64bf7a`) — SlotSelectionDialog had `onClick` on the row div AND `onCheckedChange` on the Checkbox; clicking the checkbox region fired both handlers, toggling twice. **Fix**: made the Checkbox visual-only with `tabIndex=-1, aria-hidden="true"` and moved the toggle to the row div via `role="checkbox"` + keyboard handler.

12. **Dialog race on rapid Book Shift clicks** (`b64bf7a`) — `key={slotDialogShift.id}` forces clean remount between different shifts.

### P2 bug discovered but NOT yet fixed at session end
13. **Resend click-tracking wrapper unreachable** — user reported email link `us-east-1.resend-clicks.com/CL0/...` returned `ERR_QUIC_PROTOCOL_ERROR`. Partial mitigation shipped in `send-email` function: `data-resend-track="false"` on all anchor tags + a `tags: [{name:"click_tracking", value:"off"}]` hint. **Action still needed**: disable click tracking at the account level in Resend Dashboard → Settings → Tracking.

### Safety net added
14. **`reconcile-shift-counters` cron** (`b64bf7a`) — nightly at 09:00 UTC. Resyncs `shifts.booked_slots` and `shift_time_slots.booked_slots` from ground truth. Protects against any future regression that leaves counters drifted.
